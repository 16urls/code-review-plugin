# 代码审查 Skill

## 步骤

0. **环境检查（必做，最先执行）**：
   - 执行 `git --version`，若命令不存在或报错，**立即停止后续步骤**，原样输出以下提示后结束本次审查：

     > ⚠️ 当前 skill 基于 git 记录进行工作，系统未检测到 git，请先完成安装并初始化（`git init`）。
     > 安装地址：https://git-scm.com/downloads

   - 若 `git --version` 成功但执行 `git rev-parse --is-inside-work-tree` 返回非真（当前目录不是 git 仓库），**立即停止后续步骤**，原样输出以下提示后结束本次审查：

     > ⚠️ 当前 skill 基于 git 记录进行工作，当前目录不是 git 仓库，请先在项目根目录执行 `git init` 并提交一次基线 commit。

   - 仅当用户在请求中**明确指定了要审查的文件路径**时，可跳过上述两条提示，直接用 `Read` 读取这些文件并进入步骤 2。

1. 确定审查范围：
   - 优先执行 `git diff HEAD` 获取未提交变更
   - 若 `git diff HEAD` 为空，再尝试 `git diff HEAD~1 HEAD` 审查最近一次 commit
   - 若用户指定了文件，则用 `Read` 读取用户提到的文件

2. 从以下维度检查代码：
   - **逻辑错误**：空指针、未处理异常、错误的边界条件
   - **安全问题**：SQL 注入、XSS、命令注入、敏感信息泄露、鉴权缺失
   - **性能问题**：N+1 查询、不必要的循环、内存泄漏、阻塞 I/O
   - **可维护性**：命名、重复代码、过长函数、不清晰的抽象

3. 输出 Markdown 表格报告：

   | 级别 | 文件:行 | 问题描述 | 修复建议 |
   |----|----|----|----|
   | 🔴/🟡/🟢 | `path:line` | … | … |

   级别定义：
   - 🔴 **严重**：Bug、安全漏洞、数据正确性风险
   - 🟡 **警告**：边界情况、并发隐患、性能瓶颈
   - 🟢 **建议**：可读性、命名、架构优化

   表格之后，针对每条 🔴 / 🟡 问题追加修复代码片段（diff 或完整片段）。

## 注意事项

- **环境前置**：步骤 0 的两条 git 提示是**硬约束**，不得改写或省略；命中其一时直接结束，不要继续输出表格
- 不审查二进制 / 生成文件（如 `*.lock`、`*.min.js`、`dist/`、`node_modules/`、`vendor/`）
- 没有可见证据时，标 🟡 而不是 🔴；不要臆测未读取的代码
- 仅基于实际读到的代码报告问题，不要凭印象编造行号

## 触发示例

- "帮我 review 这段代码"
- "检查 security 问题"
- "review my last commit"
- "audit this file for bugs"
